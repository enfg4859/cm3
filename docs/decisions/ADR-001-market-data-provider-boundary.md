# ADR-001: 시장 데이터는 Nitro 서버 프록시 뒤에 둔다

## 배경
대시보드는 실데이터 또는 모의 데이터를 사용해야 하고, 지표 계산은 서버에서 수행되어야 하며, 공급자 API 키도 안전하게 다뤄야 한다.
프론트엔드가 외부 공급자를 직접 호출하면 비밀값이 노출되고, 공급자별 응답 형식이 UI에 새어 나오며, mock 기반 테스트도 어려워진다.

## 결정
아래 경계를 엄격히 유지한다.
- 프론트엔드 UI: `src/*`
- Nitro API 프록시 라우트: `server/routes/api/*`
- 공급자 인터페이스 및 구현체: `server/providers/*`

모든 공급자 응답은 공유 내부 계약으로 정규화하고, 기술 지표 계산은 서버에서 수행한다.
공급자 API 키에 접근할 수 있는 계층은 Nitro 서버뿐이다.

## 고려한 대안
- 프론트엔드가 Twelve Data를 직접 호출하는 방식
  - API 키가 노출되고, UI가 공급자 응답 형식에 강하게 결합되므로 기각했다.
- 공급자 인터페이스 없이 단일 하드코딩 공급자만 두는 방식
  - mock/live 전환과 테스트가 어려워지므로 기각했다.
- 클라이언트에서 지표를 계산하는 방식
  - 뷰마다 로직이 중복되고 서버 계약 경계가 약해지므로 기각했다.

## 결과 및 영향
- 비밀값이 서버에만 남는다.
- UI는 공급자에 덜 종속적으로 유지된다.
- mock과 live 공급자가 같은 정규화 응답 형식을 공유한다.
- 기본 테스트를 `mock` 기준으로 실행해 실데이터 크레딧 소모를 막을 수 있다.
- 서버 계층이 캐시와 공급자 호출 한도 보호 책임을 더 많이 진다.
