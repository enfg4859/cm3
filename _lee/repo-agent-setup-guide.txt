바이브 코딩용 Codex 프롬프트 및 AGENTS.md 세팅 가이드

이 파일은 저장소를 에이전트 친화적으로 세팅하기 위한 기준 문서다.
에이전트는 이 문서를 따라 폴더 구조, 운영 문서, 규칙 문서를 세팅하면 된다.
목표는 다음 4가지를 강제하는 것이다.

1. 작업 규칙이 흔들리지 않게 만들 것
2. 현재 명세와 목표 상태를 분리할 것
3. 중요한 의사결정 이유를 기록할 것
4. 실제 변경 이력을 누락 없이 남길 것

==================================================
1. 핵심 원칙
==================================================

Codex 프롬프트는 얇게 둔다.
AGENTS.md에는 오래 가는 규칙만 넣는다.
실제 기능 내용은 spec 문서로 분리한다.
왜 바뀌었는지는 decision log에 남긴다.
무엇이 바뀌었는지는 changelog에 남긴다.

역할 분리는 아래처럼 본다.

- prompt = 이번 세션의 임무
- AGENTS.md = 항상 지켜야 하는 규칙
- spec = 지금 만들어야 하는 정답
- decision log = 왜 그렇게 정했는지
- changelog = 실제로 무엇을 바꿨는지

==================================================
3. 각 문서의 역할
==================================================

[1] AGENTS.md
넣을 것
- 코딩 규칙
- 테스트 규칙
- 문서 업데이트 규칙
- 변경 범위 통제 규칙
- 출력 형식
- 브랜치/커밋/PR 원칙
- 금지사항

넣지 말 것
- 개별 기능 상세 명세
- 일회성 작업 지시
- 자주 바뀌는 TODO

[2] spec 문서
넣을 것
- 문제 정의
- 목표
- 비목표
- 사용자 시나리오
- 입력/출력
- 예외 케이스
- 수용 기준
- 영향 받는 화면/API/테이블

[3] decision log
넣을 것
- 왜 이 방식을 택했는지
- 대안은 무엇이었는지
- 포기한 선택지는 무엇인지
- 장단점과 후속 영향

[4] changelog
넣을 것
- 언제
- 무엇을
- 왜
- 어떤 파일을
- 어떤 테스트와 함께 바꿨는지

[5] worklog
선택 문서다.
세션 단위 작업 메모다.
에이전트나 다음 작업자가 맥락을 이어받기 쉽게 만든다.

==================================================
4. 반드시 지켜야 할 운영 규칙
==================================================

1. AGENTS.md는 짧아야 한다.
너무 길면 안 읽는다. 오래 유지할 규칙만 넣는다.

2. spec은 현재 상태와 목표 상태를 분리해야 한다.
기존 동작과 바꿀 동작이 섞이면 에이전트가 잘못 구현한다.

3. changelog는 코드 변경과 연결되어야 한다.
"수정함" 같은 표현은 금지한다.
무엇이 어떻게 달라졌는지와 사용자 영향이 보여야 한다.

4. decision log가 없으면 같은 논쟁을 반복한다.
바이브 코딩에서 가장 자주 무너지는 지점이다.

5. 문서 업데이트를 완료 조건에 넣어야 한다.
코드만 바꾸고 문서를 안 바꾸면 바로 드리프트가 발생한다.

==================================================
5. Codex 프롬프트 기본 틀
==================================================

아래 파일을 prompts/codex-task-template.md 같은 이름으로 둔다.
세션 시작 시 작업 프롬프트의 기본 뼈대로 사용한다.

--------------------------------------------------
You are the coding agent for this repository.

Your job:
- Implement only the requested scope.
- Follow repository rules in AGENTS.md first.
- Read the relevant spec before changing code.
- Make the smallest complete change that satisfies the spec.
- Do not silently expand scope.

Priority order:
1. User task
2. AGENTS.md
3. docs/specs/current/*
4. docs/decisions/*
5. existing code patterns

Required workflow:
1. Read relevant files before editing.
2. Summarize the task in 3-5 lines.
3. State assumptions if the spec is incomplete.
4. Make a minimal implementation plan.
5. Implement.
6. Run or describe relevant tests.
7. Update docs if behavior/contract changed:
   - spec if expected behavior changed
   - decision log if architecture/tradeoff changed
   - changelog for meaningful code changes
8. Report:
   - files changed
   - behavior changed
   - tests run
   - risks / follow-up items

Non-negotiable rules:
- Do not rewrite unrelated files.
- Do not introduce new dependencies unless necessary.
- Do not change naming conventions or architecture without recording it.
- Prefer explicitness over cleverness.
- Prefer reversible changes.
- Preserve backward compatibility unless spec says otherwise.

Definition of done:
- Code matches spec
- Tests added/updated as needed
- Docs synced
- No hidden scope expansion
--------------------------------------------------

이 프롬프트의 핵심은 아래 3개다.
- 우선순위
- 작업 절차
- 완료 조건

==================================================
6. AGENTS.md 기본 틀
==================================================

아래를 기본 골격으로 사용한다.
저장소 규칙에 맞게 필요한 부분만 추가한다.

--------------------------------------------------
# AGENTS.md

## 1. 목적
이 저장소의 에이전트는 요구사항에 맞는 최소 변경을 수행하고, 문서와 코드를 항상 동기화한다.

## 2. 우선순위
작업 우선순위는 다음과 같다.
1. 명시된 사용자 요청
2. 이 문서의 규칙
3. docs/specs/current/*
4. docs/decisions/*
5. 기존 코드 패턴

## 3. 작업 원칙
- 변경 범위를 임의로 넓히지 않는다.
- 먼저 읽고, 그다음 수정한다.
- 최소 변경으로 문제를 해결한다.
- 추측이 필요한 경우 가정을 명시한다.
- 동작 변경이 있으면 문서를 함께 수정한다.
- 되돌리기 어려운 변경은 피한다.

## 4. 코드 작성 규칙
- 프론트엔드는 camelCase를 사용한다.
- 백엔드는 snake_case를 사용한다.
- 기존 구조와 네이밍을 우선 따른다.
- 불필요한 추상화와 과도한 리팩터링을 피한다.
- 주석은 필요한 경우에만 작성하고, 코드가 더 명확하도록 우선 수정한다.

## 5. 테스트 규칙
- unit test는 Jest를 사용한다.
- E2E는 Maestro를 사용한다.
- 동작이 바뀌면 관련 테스트를 추가 또는 수정한다.
- 테스트가 불가능하면 이유와 대체 검증 방법을 보고한다.

## 6. 문서 규칙
동작, 계약, 사용자 경험, 구조 결정이 바뀌면 아래 문서를 함께 갱신한다.
- spec: 기대 동작 변경 시
- decision log: 구조/트레이드오프 변경 시
- changelog: 의미 있는 코드 변경 시

## 7. Spec 관리 규칙
각 기능 spec은 아래 항목을 포함한다.
- 배경
- 목표
- 비목표
- 범위
- 사용자 시나리오
- 입력/출력
- 예외/실패 케이스
- 수용 기준
- 영향 받는 모듈/API/UI

spec이 불완전하면 에이전트는 임의 확장 대신 가정을 명시하고 최소 범위만 구현한다.

## 8. Decision log 규칙
중요한 기술적 선택은 ADR 형식으로 기록한다.
반드시 아래를 포함한다.
- Context
- Decision
- Alternatives considered
- Consequences

## 9. Changelog 규칙
모든 의미 있는 변경은 changelog에 기록한다.
반드시 아래를 포함한다.
- 날짜
- 변경 요약
- 이유
- 영향 범위
- 관련 파일
- 테스트 상태

## 10. 금지사항
- 관련 없는 파일 대량 수정
- 명세 없는 광범위 리팩터링
- 문서 미갱신 상태로 종료
- 테스트 없이 동작 변경 마감
- 이유 없는 의존성 추가

## 11. 완료 보고 형식
작업 완료 시 아래 형식으로 보고한다.
- Summary
- Files changed
- Behavior changes
- Tests
- Docs updated
- Risks / follow-ups
--------------------------------------------------

10. 추천 작업 루프
==================================================

1. 작업 시작
- spec 확인
- 관련 ADR 확인
- AGENTS.md 확인

2. 구현 전
- 변경 범위 확정
- 가정 명시

3. 구현
- 작은 단위로 수정
- 기존 패턴 존중

4. 구현 후
- 테스트 수행
- spec/doc 동기화
- changelog 기록

5. 종료
- 무엇을 바꿨는지
- 왜 바꿨는지
- 어떤 리스크가 남았는지 보고

==================================================
11. 바이브 코딩에서 자주 망하는 패턴
==================================================

- AGENTS.md 하나에 모든 내용을 몰아넣음
- spec 없이 코드만 보고 수정함
- 왜 바꿨는지 기록이 없음
- 변경 로그가 너무 빈약함
- 완료 조건이 없음
- "깔끔하게 리팩터링" 같은 모호한 지시를 줌

실전 비유로 보면 아래가 맞다.

- AGENTS.md = 헌법
- spec = 현재 법률
- decision log = 판례
- changelog = 사건 기록
